GPU-Accelerated Web Applications Proposal

Overview

This document proposes four interactive web applications that demonstrate the computational advantages of GPU architectures over traditional CPU processing. Each project leverages CUDA acceleration through CuPy and Numba for server-side computation, real-time data streaming via Socket.IO, and Three.js for hardware-accelerated client-side visualization.

Project 1: N-Body Gravitational Simulation

Summary

An interactive 3D visualization of gravitational interactions between celestial bodies, demonstrating massive parallelization advantages for pairwise force calculations.

Technical Objectives

  • Implement gravitational N-body simulation supporting 1,000 to 1,000,000 particles
  • Compare CPU sequential force calculation (O(n²)) with GPU parallel implementation
  • Achieve real-time visualization at 30-60 FPS with interactive camera controls
  • Display live performance metrics showing computation time and speedup ratios

Computational Challenge

Each particle experiences gravitational force from every other particle, requiring n×(n-1)/2 pairwise calculations per timestep. For 100,000 particles, this equals approximately 5 billion force calculations per frame.

Technology Stack

Server-Side Processing:

  • CuPy: Vectorized pairwise distance calculations, force accumulation using GPU array operations
  • Numba CUDA: Custom kernels for optimized memory access patterns, reducing global memory transactions
  • FastAPI: RESTful endpoints for simulation configuration and initial setup
  • python-socketio: Real-time streaming of particle state (positions, velocities) at target frame rate

Client-Side Visualization:

  • Socket.IO Client: Bidirectional communication for parameter updates and state streaming
  • Three.js: Particle rendering using THREE.Points with BufferGeometry for efficient updates, orbit controls for 3D navigation, optional motion trails and bloom effects

Key Features

  • Adjustable particle count with real-time performance impact
  • Preset scenarios (binary stars, galaxy collision, solar system)
  • Interactive spawn mode for adding particles during simulation
  • Side-by-side CPU vs GPU performance comparison
  • Configurable physics parameters (gravitational constant, softening factor, timestep)

Success Metrics

  • 100K particles at 30+ FPS on GPU vs <1 FPS on CPU
  • Demonstrable 50-100x speedup for large particle counts
  • Stable orbital mechanics without numerical divergence
  • Smooth visualization without frame drops

Project 2: 2D Fluid Simulation (Stable Fluids)

Summary

Real-time smoke and fluid dynamics solver implementing the Stable Fluids algorithm, showcasing GPU acceleration of iterative grid-based numerical methods.

Technical Objectives

  • Implement incompressible Navier-Stokes solver with advection, diffusion, and pressure projection
  • Compare iterative solver performance (Jacobi/Gauss-Seidel methods) between CPU and GPU
  • Enable interactive fluid manipulation through mouse/touch input
  • Visualize velocity fields and density distribution with artistic rendering options

Computational Challenge

Each simulation step requires solving sparse linear systems iteratively across the entire grid. For a 512×512 grid with 50 Jacobi iterations, this involves 13 million operations per frame. Additionally, advection requires bilinear interpolation at each grid point.

Technology Stack

Server-Side Processing:

  • CuPy: Grid operations for advection (semi-Lagrangian method), diffusion kernels, linear system solving
  • Numba CUDA: Optimized stencil operations for neighbor calculations, shared memory utilization for border handling, pressure projection with reduced global memory access
  • FastAPI: Static file serving and simulation parameter configuration
  • python-socketio: Bidirectional streaming of grid state (density, velocity fields) and mouse interaction coordinates

Client-Side Visualization:

  • Socket.IO Client: Stream user input events (click position, drag velocity) and receive grid updates
  • Three.js: Density field rendered as THREE.DataTexture on textured plane, velocity visualization using arrow helpers or line segments, optional volumetric rendering for 3D smoke effects, custom fragment shaders for artistic color mapping

Key Features

  • Multiple fluid injection modes (smoke, dye, force application)
  • Adjustable grid resolution (64² to 1024²) with performance scaling
  • Boundary condition options (wrap-around, solid walls, open boundaries)
  • Visualization modes (density only, velocity vectors, divergence field)
  • Configurable physical parameters (viscosity, diffusion rate, timestep)
  • Recording and playback of interaction sequences

Success Metrics

  • 512² grid at 60 FPS on GPU vs <5 FPS on CPU
  • Visually smooth, artifact-free smoke behavior
  • Demonstrable 10-20x speedup for iterative solver operations
  • Responsive interaction with <50ms latency from input to visual feedback

Project 3: 3D Cellular Automata Explorer

Summary

Extension of Conway's Game of Life into three-dimensional space, demonstrating GPU optimization of stencil operations and emergent complexity visualization.

Technical Objectives

  • Implement 3D cellular automata with configurable survival/birth rules
  • Process large grids (256³ to 512³ cells) with efficient boundary handling
  • Demonstrate shared memory optimization for neighbor counting operations
  • Explore and visualize emergent 3D patterns and structures

Computational Challenge

Each cell requires counting 26 neighbors in 3D space (compared to 8 in 2D). For a 512³ grid, this involves examining 3.4 billion neighbor relationships per generation. Naive implementations suffer from excessive global memory access.

Technology Stack

Server-Side Processing:

  • CuPy: 3D convolution for neighbor counting using separable kernels, state update logic
  • Numba CUDA: Custom kernels with shared memory tiles to minimize memory bandwidth requirements, optimized boundary condition handling (periodic, fixed, reflective)
  • FastAPI: Pattern library management, rule configuration endpoints
  • python-socketio: Stream alive cell coordinates or compressed grid deltas using run-length encoding

Client-Side Visualization:

  • Socket.IO Client: Receive incremental grid updates and pattern spawn commands
  • Three.js: Alive cells rendered as THREE.InstancedMesh (cubes or spheres), frustum culling and LOD for performance with large populations, camera controls with smooth transitions, optional cell age coloring and transparency effects

Key Features

  • Preset patterns (3D gliders, oscillators, still lifes)
  • Custom rule designer (survival/birth neighbor count ranges)
  • Grid cross-section viewer for internal structure analysis
  • Pattern seeding tools (random, symmetric, imported)
  • Generation step control (single-step, continuous, variable speed)
  • Population statistics tracking (alive cells, birth rate, stability detection)

Success Metrics

  • 256³ grid processing at 10+ generations/second on GPU vs <0.1 gen/sec on CPU
  • Demonstrable 50-100x speedup from shared memory optimization
  • Stable long-term evolution without memory leaks
  • Discovery and documentation of interesting 3D emergent patterns

Project 4: Progressive Ray Tracer

Summary

Monte Carlo path tracing renderer demonstrating embarrassingly parallel ray generation and intersection testing, with progressive quality refinement over time.

Technical Objectives

  • Implement physically-based path tracing with multiple material types
  • Compare single-threaded CPU ray tracing with massively parallel GPU execution
  • Progressive tile-based rendering showing quality improvement over iterations
  • Support for interactive camera movement with automatic restart

Computational Challenge

Path tracing requires casting millions of rays per pixel (typically 1000+ samples for noise-free images). Each ray involves multiple recursive bounces with intersection tests against scene geometry. A 1920×1080 image at 1000 samples requires tracing over 2 billion primary rays.

Technology Stack

Server-Side Processing:

  • CuPy: Ray-sphere/plane intersection calculations, material BRDF evaluation, random number generation for Monte Carlo sampling
  • Numba CUDA: Custom ray tracing kernel with optimized ray generation, BVH (Bounding Volume Hierarchy) traversal for complex scenes, importance sampling for materials (cosine-weighted hemisphere, microfacet distributions)
  • FastAPI: Scene description API, camera configuration, material library
  • python-socketio: Stream rendered tiles (8×8 or 16×16 pixel blocks) with progressive sample accumulation

Client-Side Visualization:

  • Socket.IO Client: Receive image tiles and composite into final display, send camera transformation updates
  • Three.js: Display progressive result as THREE.CanvasTexture updated per tile, optional 3D wireframe preview of scene geometry for positioning, orbit/pan controls for camera manipulation, real-time sample count and convergence display

Key Features

  • Multiple material types (Lambertian diffuse, metallic, dielectric/glass, emissive)
  • Configurable scene complexity (primitives: spheres, planes, triangles)
  • Tile prioritization (center-out, importance-based)
  • Sample accumulation with variance estimation
  • Render quality presets (preview, medium, high, production)
  • Scene templates (Cornell box, glass spheres, metallic objects)
  • Export rendered images at completion

Success Metrics

  • 1920×1080 preview (100 samples) in <30 seconds on GPU vs 10+ minutes on CPU
  • Demonstrable 100-500x speedup for large sample counts
  • Visually correct materials (color bleeding, caustics, reflections)
  • Progressive updates showing smooth quality improvement
  • Noise-free convergence at high sample counts (1000+)

Common Architecture

Server Infrastructure 

FastAPI Application
├── RESTful API (initial configuration, file uploads)
├── python-socketio (WebSocket-based real-time communication)
├── CuPy/Numba Compute Layer (CUDA kernel execution)
└── Performance Monitoring (CPU/GPU timing, memory usage)

Client Infrastructure 

Single-Page Application
├── Socket.IO Client (bidirectional communication)
├── Three.js Renderer (WebGL-accelerated visualization)
├── UI Controls (parameter adjustment, presets)
└── Performance Dashboard (FPS, latency, speedup metrics)

Data Flow

  1. User configures simulation parameters via web interface
  2. Client sends configuration to server via Socket.IO
  3. Server initializes GPU computation and begins processing
  4. Server streams results (binary or JSON) at target frame rate
  5. Client updates Three.js scene with received data
  6. Client sends interaction events (mouse, camera) back to server
  7. Server adjusts computation based on interaction

Performance Monitoring

All projects include real-time display of:

  • CPU computation time per frame
  • GPU computation time per frame (including memory transfers)
  • Speedup ratio (CPU time / GPU time)
  • Frame rate and frame time distribution
  • Memory usage (host and device)
  • Data transfer bandwidth

Implementation Phases

Phase 1: Core Infrastructure

  • FastAPI + python-socketio server setup
  • Socket.IO client integration with Three.js
  • Basic GPU kernel scaffolding with CuPy
  • Performance timing and metrics collection

Phase 2: CPU Baseline

  • Implement correct CPU reference versions for each project
  • Validate numerical accuracy and stability
  • Establish baseline performance measurements
  • Create minimal visualization for correctness verification

Phase 3: GPU Implementation

  • Port algorithms to CuPy vectorized operations
  • Optimize with custom Numba CUDA kernels
  • Implement memory transfer optimization (pinned memory, async transfers)
  • Profile and optimize hotspots

Phase 4: Visualization Enhancement

  • Implement full Three.js rendering features
  • Add interactive controls and parameter adjustment
  • Create preset configurations and demos
  • Implement recording and playback capabilities

Phase 5: Documentation and Refinement

  • User documentation and tutorials
  • Code documentation and architecture diagrams
  • Performance benchmarking across hardware configurations
  • Bug fixes and polish

Technical Requirements

Hardware

  • NVIDIA GPU with CUDA Compute Capability 6.0+ (Pascal architecture or newer)
  • 8GB+ GPU memory recommended for large simulations
  • 16GB+ system RAM
  • Modern web browser with WebGL 2.0 support

Software Dependencies

Backend:

  • Python 3.9+
  • CUDA Toolkit 11.x or 12.x
  • CuPy (matching CUDA version)
  • Numba with CUDA support
  • FastAPI
  • python-socketio
  • uvicorn (ASGI server)

Frontend:

  • Node.js 18+ (for development tooling)
  • Three.js r150+
  • Socket.IO client
  • Modern build toolchain (Vite/Webpack)

Conclusion

These four projects provide diverse demonstrations of GPU computing advantages across different problem domains: O(n²) pairwise interactions, iterative numerical solvers, stencil operations with locality optimization, and embarrassingly parallel ray casting. Each project combines rigorous computational work with engaging visual output, making the performance benefits immediately apparent to users.

The unified architecture (FastAPI + Socket.IO + Three.js) provides a consistent foundation while allowing each project to showcase unique aspects of GPU programming. Real-time interaction and progressive visualization make the applications both educational and enjoyable to use.
